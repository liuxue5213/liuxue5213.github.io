php_codesniffer配置（代码规范提示）

打开CMD，进入到需要安装PHP_CodeSniffer的目录，例如E:\xampp\php\phar（可以是任意目录，但是最好选择一个固定的目录，用于安装php的第三方的库文件），执行如下命令

composer require "squizlabs/php_codesniffer=*"

该命令会在当前目录下生成vendor目录、composer.json文件、composer.lock文件，安装的库文件在vendor目录下，

然后下载 symfony2的标准文件到E:\xampp\php\phar\vendor\squizlabs\php_codesniffer\CodeSniffer\Standards目录下解压

然后在PHPstorm的settings中搜索sniffer，找到路径配置，将路径设置为E:\xampp\php\phar\vendor\bin\phpcs.bat，点击[Validate]，

如果提示版本信息则说明配置正确

在inspections中找到PHP_CodeSniffer validation，将编码规范配置为symfony2即可

版本控制

版本控制使用git，在windows下的客户端使用Tortoisegit，安装Tortoisegit之前先安装 Git for windows ，

安装完成后安装Tortoisegit 点此下载，然后再安装 中文语言包，

服务器端使用gitlab，访问地址为http://git.wintels.cn，每个人都分配单独的账号登陆，工作流程如下：

  1. 项目经理创建项目的版本库
  2. fork项目经理创建的版本库到自己的名下
  3. 将自己名下的版本库clone到本地开发环境
  4. 开发完成后先提交到本地，然后推送到自己名下的版本库中
  5. 在自己名下的版本库中创建Merge Requests请求合并到项目经理的版本库对应的分支
  6. 项目经理review代码后选择合并或关闭合并请求
  7. 重复步骤4-6

编码规范

该规范只适合于symfony框架，原CI框架可不按照此规范

例子

该规范采用自symfony编码规范，该规范在 PSR-1和 PSR-2 基础之上，先仔细查看这两个规范，然后再查看如下规范

<?php

/*
 * This file is part of the Symfony package.
 *
 * (c) Fabien Potencier <fabien@symfony.com>
 *
 * For the full copyright and license information, please view the LICENSE
 * file that was distributed with this source code.
 */

namespace Acme;

/**
 * Coding standards demonstration.
 */
class FooBar
{
    const SOME_CONST = 42;

    private $fooBar;

    /**
     * @param string $dummy Some argument description
     */
    public function __construct($dummy)
    {
        $this->fooBar = $this->transformText($dummy);
    }

    /**
     * Transform text（说明函数的用途）
     *（说明和参数之间必须空一行）
     * @param string $dummy   Some argument description
     * @param array  $options Some options（参数说明，每个参数都需要说明类型和用途，且多个参数的名称和说明都需要上下对齐）
     *（参数和@return之间必须空一行）
     * @return string|null Transformed input（返回值说明）
     *（@throws和@return之间也必须空一行）
     * @throws \RuntimeException（抛出异常说明）
     */
    private function transformText($dummy, array $options = array())
    {
        $mergedOptions = array_merge(
            array(
                'some_default' => 'values',
                'another_default' => 'more values',（array中的每个元素后面必须包含逗号，最后一个元素也不例外）
            ),
            $options
        );

        if (true === $dummy) {
            return;（语句块中如果只有一个return语句则不需要空行）
        }
        （多个if等语句块之间最好加空格）
        if ('string' === $dummy) {
            if ('values' === $mergedOptions['some_default']) {
                return substr($dummy, 0, 5);
            }

            return ucwords($dummy);（语句块中存在其他语句的情况下，return之前必须要有一个空行）
        }
            （空一行）
        throw new \RuntimeException(sprintf('Unrecognized dummy option "%s"', $dummy));
    }

    private function reverseBoolean($value = null, $theSwitch = false)
    {
        if (!$theSwitch) {
            return;
        }

        return !$value;
    }
}

文件结构

  • 每个逗号后面添加一个空格
  • 操作符（==,&&,…）等两边各添加一个空格，操作符（.）除外
  • 一元操作符（！，–，。。。）等跟变量之间不加空格，如!$num
  • 多行数组的每个变量的后面添加一个空格，包括最后一个元素
  • return语句上面添加一个空行，除了只有return语句的语句块
  • 语句块结构必须包含在{}中，不管该语句块结构中包含多少行语句
  • 每个文件只定义个class
  • class中的属性定义在方法前面
  • class中方法的定义顺序为public，protected，private
  • 实例化class时，不管构造函数有多少个参数，都加上花括号()
  • 异常信息的输出使用sprintf函数来格式化

命名规范

  • 所有的变量、方法、函数、参数名字都以“骆驼命名”方式命名，不使用下划线的方式
  • 除上面外的数组的下标名、静态变量名（大写）、配置参数等以下划线的方式命名
  • 所有的class都定义在namespace下
  • 抽象类命名都以Abstract开头
  • 接口命名以Interface结尾
  • 异常类命名以Exception结尾
  • 对文件名使用字母数字字符以及下划线
  • PHPDocs中的变量的类型使用：bool布尔类型、int整数类型、float浮点数类型、array数组类型、stdClass对象类型，如果是某个确定的类的对象，则使用确定的类
  • 其他的参考PSR-1和PSR-2

Service名字命名规范

  • service名字使用bundle的名字开头，用于分组使用，如icsoc_menu.list
  • service名字使用小写字符
  • service名字使用下划线分隔

PHP注释规范

  • 所有的类、方法、函数都添加PHPDocs，PHPstorm中添加的方法为输入/**后直接按回车会自动生成类、方法的注释
  • 如果方法没有返回值，则在注释中去掉@return项
  • 方法注释中所有的参数都需要加上类型和备注说明，具体看上面例子
  • 方法注释中备注说明必须在最上面（如果是一看就能明白的函数可以不加备注说明，比如构造函数），然后下面空一行，再是参数的说明，@return上添加一个空行
